In plain English

Think of it as the perfect temp clerk

Hire a temp on Monday and hand them insurance forms they've never seen — different layouts, different wording, boxes in a different order on every page. A great clerk doesn't memorize forms. They read each box, work out what it's actually asking for, and pull the right detail from the applicant's file.

FormWright is that clerk. You give it one file of answers about the applicant, point it at a live web form it has never encountered, and it fills the form out — page by page — with no cheat sheet for that particular form baked in.

And here's the part that makes it trustworthy: when it isn't sure, it doesn't guess. If a form asks for "County" and the applicant's file has no county, a bad clerk scribbles something plausible. FormWright leaves the box empty and leaves a sticky note — "couldn't answer this one." An empty, flagged box is honest. A confidently wrong answer on an insurance application is a real problem.

One rule is the whole personality of the system: understand every box, never fake an answer.
One box, start to finishWhat happens
1 · See the boxRead the label, the header above it, the text beside it.
2 · Read its meaning"Entity Type" → the business's legal structure.
3 · Find the answerThe file says the business is an LLC.
4 · Format to fitThis box wants the full name: "Limited Liability Company".
5 · Type it inFilled — via a handle the agent alone controls.
6 · Verify it stuckRe‑read the box. Really in there? Move on.

If at any step it isn't confident enough, it stops, leaves the box blank, and flags it. No bluffing.

Why it's different

Six ideas that separate it from a form‑filling script

🧭

Generalizes to an unseen form

The graded run is a hidden third form — no selectors, ids, or labels known in advance. Brittle scripts break when a field moves; FormWright doesn't script the form, it reads the page. Proven on an adversarial form and a live public one.

🛑

Abstains instead of guessing

A blank field is a follow‑up email; a wrong field on an application is a mispriced policy. The costs aren't symmetric, so it never averages them away. Hallucinated fills across every test form: zero.

💵

Safety that costs $0

The cascade runs cheapest‑first; the deterministic core clears every field it should for $0, ~6s/form, zero LLM calls, no API key. Cost never erodes safety — the safe path is the free path.

🔒

A grounded LLM that can't hallucinate

When used, the model is a bounded classifier over a closed set: it may only pick an existing key (or no_answer) and an existing option. It selects; it cannot compose. No model words ever reach the form.

🧬

Adaptive & self‑learning

It reads any form's structure at runtime, so the same code runs on a form it's never seen. A cheap‑to‑expensive cascade spends effort only where a field needs it — and it gets faster with use: once it works out a carrier's form it caches those mappings, so every form from that carrier after the first resolves instantly and free. It learns a carrier simply by using it.

🔬

Auditable by construction

Every run writes one record per field — label, binding method, chosen key, the full candidate scoreboard, the value, and the verify result. Every decision, including every abstention, is defensible.

How it works

One closed loop, driven on every page

Nothing about the target form is known in advance — each step earns the next. The same loop runs whether it's a sample form or a hidden one.

Perceive

Enumerate every visible control; compute its accessible name (aria, label[for], wrapping labels, <th> headers, geometry); stamp a code‑owned handle.

Presence gate

Could any answermap key plausibly fill this field? If not — abstain now. No key, no guess.

Map — cascade

exact → alias/concept → fuzzy → embeddings → grounded LLM. Stop at the first confident hit.

Format

Deterministic renderers keyed off type/pattern/placeholder: (619) 555‑0142 → 6195550142.

Act

Fill / select / check via Playwright — through the owned handle, never a brittle selector.

Read‑back verify

Re‑read the DOM and assert the value landed. If it didn't, it didn't happen.

Repeat & reveal

Expand repeating rows and conditional fields, then feed them back through the same loop.

Classify transition

Confirmation / error markers + a step‑signature diff. Never assumes the URL changed.

Gate on both, always: a field is filled only when combined = labelConfidence × matchScore clears the threshold. Below the line, it stays blank and flagged — abstention is a first‑class result, not a failure.

The ML inside

A matching problem, not a generation problem

That single design choice is why an LLM can sit in the loop and still produce zero hallucinated fills. The layers work cheapest‑first, most‑powerful‑last.

Why not a vision model?

The browser already exposes the ground truth. The DOM and accessibility tree are exact, cheap, and directly actionable — a resolved aria-labelledby or <th> header is the field's real name. A vision model would re‑derive that approximately, at higher cost, with a fresh class of perception errors. Why infer what you can read exactly?

Why form‑transformers were rejected

LayoutLM / MarkupLM need fine‑tuning on labeled form data — and the whole brief is a form the agent has never seen, graded on a hidden third. A model that must be trained on a distribution it will never get is disqualified by definition. FormWright composes small, deterministic, inspectable parts that work on the first encounter.

The algorithms inside

Classical algorithms — chosen deliberately, and chosen against

Filling a page is an assignment problem, not a search: fields on the left, answermap keys on the right, each pairing weighted by combined = labelConfidence × matchScore. Find the jointly‑best assignment — without two fields fighting over one key.

Greedy fast‑path → Hungarian on contention

Full optimal assignment is the Hungarian algorithm (Kuhn–Munkres), O(n³). Paying that on every page is wasteful — most fields have one obvious home. So: a greedy fast‑path takes each field's top key; if nothing contends, that's already optimal. The moment two fields want the same key, the contested cluster is handed to exact Hungarian — cubic cost paid only over the fields actually in dispute.

Fieldbest key2nd‑best
"Insured Name"full_name .91business_name .55
"Business Name"full_name .88business_name .86

Greedy hands both fields full_name — a collision. Hungarian maximizes the sum: → full_name (.91) + → business_name (.86) = 1.77. It never double‑books a key or fabricates a value to paper over the clash.

The qualifier penalty

Insurance forms are full of near‑duplicate keys differing only by a qualifier — current_ vs prior_, driver_1 vs driver_2. Similarity sees them as identical and lets a prior‑term field steal the current‑term key. A qualifier penalty docks the weight when a candidate and the field disagree on a discriminating qualifier — keeping the assignment honest exactly where two keys are otherwise indistinguishable.

Min‑heap top‑k ranking

Under every field is a ranked shortlist of candidate keys, maintained with a min‑heap / top‑k at O(n log k) instead of a full sort. The same structure feeds the confidence check and the audit trail: what it chose, what it almost chose, and by how much.

Column‑order‑agnostic row chunking

Repeating grids arrive as a flat run of controls with no reliable row wrapper. Rows are reconstructed by duplicate‑key boundary detection — when a key already seen in the current chunk reappears, a new row begins. Because boundaries come from repetition, not fixed positions, it survives shuffled columns and the adversarial "no‑add‑button, description‑first" grid.

Evaluated and rejected — on purpose

  • Gale–Shapley (stable matching) — solves the wrong objective. We want maximum‑weight assignment, not a stability equilibrium. Stability ≠ optimality.
  • Dijkstra / A* — page navigation is a trivial linear walk; there's no weighted graph to shortest‑path over. The transition is a signature diff, not a search.
  • Min‑cost max‑flow — overkill for one‑to‑one matching; it earns its place only for capped one‑to‑many. Hungarian is the exact, simpler tool here.

Adaptive · self‑learning

Point it at a form it has never seen. It just works.

FormWright carries no per‑form code and no hardcoded selectors. It reads any form's structure at runtime and runs unchanged on the next one — new labels, unfamiliar synonyms, unusual widgets, repeating rows, and conditional reveals all handled on first contact, with no training and no labeled data.

It spends effort only where a field demands it. A cheap‑to‑expensive cascade escalates one field at a time — an exact match resolves for free, then a curated domain concept table, then fuzzy matching, then local embeddings, and only a genuinely ambiguous field reaches the grounded LLM. Easy fields stay instant and free; hard fields get exactly as much reasoning as they need, never a drop more.

And it gets sharper the more you use it. The first time it meets a carrier, it works out that carrier's field‑to‑key mappings and value formatters and caches them, keyed to the form's structural signature. Every run after, those learned mappings resolve instantly and for free — so a carrier you process often becomes faster and cheaper over time. It learns a carrier simply by being used on it.

What makes it adaptive

  • Zero per‑form code — reads any form's structure live; runs unchanged on the unseen.
  • Accessible‑name resolution — derives each field's meaning from ARIA, labels, <th> headers, sibling text, or geometry.
  • Effort that scales to the field — exact → alias → fuzzy → embeddings/k‑NN → grounded LLM, only as far as needed.
  • Absorbs the unseen — new labels, synonyms, and widgets, with no rule changes.
  • Real‑world mechanics — repeating rows, conditional reveals, value reformatting — automatically.
  • Learns each carrier by use — caches resolved mappings so repeat forms resolve instantly, at zero cost.
  • No retraining, ever — generalizes on first encounter; improves free with usage.

Get started

Running in under a minute

Clone it, install once, point it at any form — a local file or a live URL. No account, no API key, no setup ceremony: the deterministic core runs entirely on your machine.

# 1 · clone & install
git clone https://github.com/AndrousStark/formwright && cd formwright
npm install
npx playwright install chromium

# 2 · run it on ANY form — local file or live URL
npx tsx src/index.ts --form <url-or-path> --answers answermap.json --out runs

# optional · enable the grounded‑LLM tier for tricky fields
export ANTHROPIC_API_KEY=...   # or OPENAI_API_KEY

One command, one JSON, any form. The same command runs on a form it has never seen — and needs no API key for the deterministic core.

What you get from every run

A complete, inspectable audit trail in your runs/ folder:

  • decisions.jsonl — one line per field: label read, key chosen, every candidate's score, value written, verify result
  • run_summary.json — the run at a glance: filled, flagged, blank, cost
  • captured.json — the exact data the form received
  • trace.zip — a full Playwright trace you can replay step by step
  • page_*.png — the form as it was actually filled, page by page
